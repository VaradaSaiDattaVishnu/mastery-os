A fully client-side, zero-backend MongoDB learning environment built with React 18, Vite, Tailwind, Framer Motion, Zustand, and a custom in-browser query engine. Two courses totaling 178 lessons (18 modules of MQL, 6 modules of MongoDB at Scale), with a syntax-highlighting shell, aggregation pipeline visualizer, B-tree index visualizer, and a scale simulator — all running entirely in the browser from static files on GitHub Pages.

## Architecture

The system has four distinct runtime layers, all client-side.

**Query engine (`src/lib/sampleData.js`).** Four hardcoded in-memory collections (`users`, `products`, `orders`, `logs`) are plain JavaScript arrays of objects. `evaluateQuery(collectionName, query)` runs `Array.filter` over the target collection, delegating per-document matching to `matchesQuery`. That function handles the logical operators `$and`, `$or`, `$nor` recursively, then dispatches field-level conditions to `matchesOperators`, which covers `$eq`, `$ne`, `$gt/$gte/$lt/$lte`, `$in/$nin`, `$exists`, `$regex/$options`, `$not`, `$size`, `$all`, `$mod`, `$type`, and `$elemMatch`. Dot-notation field access (`address.city`) is resolved by `getNestedValue` via `path.split('.').reduce`. Array fields queried with a scalar use element-inclusion semantics matching real MongoDB behavior.

**Shell (`src/components/interactive/MongoShell.jsx`).** A `<textarea>` overlaid by a syntax-highlighted `<div>` (scroll-synced via `onScroll`). User input is parsed with a regex: `db\.(\w+)\.(\w+)\(([\s\S]*)\)`. Arguments are first cleaned (unquoted keys patched to quoted JSON, single-to-double-quote swap, trailing-comma removal) and passed to `JSON.parse`; on failure, a `new Function('return ' + argsStr)()` fallback handles JS expressions like regex literals. Supported methods are `find`, `findOne`, `countDocuments`, and `distinct`. Shell history (last 100 entries) is persisted to `localStorage` via Zustand's `persist` middleware under the key `mongo-mastery-storage`.

**Syntax highlighter (`src/lib/syntaxHighlight.js`).** A hand-written single-pass tokenizer (no regex engine, no external library). It walks the source character by character, emitting tokens of types `comment`, `string`, `number`, `operator` (`$`-prefixed known operators), `field`, `keyword`, `method`, and `bool`. About 130 MongoDB operators are registered in a static array and matched by exact string identity. `getTokenClass` maps token types to Tailwind class names applied in the overlay div.

**Pipeline visualizer (`src/components/interactive/PipelineVisualizer.jsx`).** A drag-free, animated stage list. Ten stage types (`$match`, `$project`, `$group`, `$sort`, `$limit`, `$skip`, `$unwind`, `$lookup`, `$addFields`, `$count`) each carry a default `spec` object shown in an expandable code block. The "Visualize" button does not execute the pipeline against data; it runs a `setInterval` at 800 ms steps, advancing `activeStage` from 0 to `stages.length - 1` and applying a `shadow-glow` border to the active `StageBlock`. The visualization is purely presentational.

**Index visualizer (`src/components/interactive/IndexVisualizer.jsx`).** A static 3-level B-tree is hard-coded: root `[30]`, two internal nodes `[10,20]` and `[40,50]`, six leaf nodes. `simulateSearch` traces a deterministic traversal path level by level (root → one internal node → one leaf), animating `highlightedPath` with `setInterval` at 500 ms. `simulateCollScan` iterates `allValues` (23 integers in order) at 100 ms per step. Stats shown: scan type (`IXSCAN`/`COLLSCAN`), docs examined (path depth ≤ 3 vs up to 23), and found/not-found.

**Scale simulator (`src/components/interactive/ScaleSimulator.jsx`).** A slider over seven scale points (1K–1B). Query time is computed as `Math.log2(docCount) * 0.5` for IXSCAN and `docCount * 0.001` for COLLSCAN, modelling O(log n) vs O(n). Docs examined for IXSCAN is `min(log2(docCount) + 1, 30)`. Animated progress bars rendered via Framer Motion `animate={{ width }}`.

**Routing and state.** React Router v6 with routes `/`, `/course/:courseId`, `/course/:courseId/:moduleId`, `/course/:courseId/:moduleId/:lessonId`. Lesson metadata lives in `courseData.js` (two courses, 24 modules, 178 lessons). Per-lesson content (HTML explanation, syntax block, examples, gotchas, challenges, starter code) lives in `lessonContent.js` and four batch files, keyed by `"courseId.moduleId.lessonId"`. `LessonView` picks the matching visualizer from `LESSON_VISUALIZERS` (a static map of nine lesson keys) and renders it inline on the Learn tab. The Monaco editor wrapper (`@monaco-editor/react`) is imported but the primary shell uses the custom textarea/overlay — Monaco is available for richer editing contexts.

## Three decisions you must justify

**Decision 1: Build a real in-browser query engine instead of canned/hardcoded outputs.**
Why: Learners need to write arbitrary queries and get real feedback. If outputs were hardcoded per lesson, any deviation from the expected query would silently return no result or a wrong one, breaking the learning loop.
Rejected alternative: Precompute correct outputs for each challenge and compare strings. This would work for guided challenges but kill free exploration — users could not experiment with operators not in the lesson.
Tradeoff: The engine covers about 20 operators and four methods. Operators like `$geoNear`, `$text`, `$expr`, or multi-key aggregation pipelines are not implemented. An unsupported operator throws `"Operator '...' isn't supported in the in-browser simulator yet"`. This is honest but it means learners can write syntactically valid MQL that the engine refuses, potentially causing confusion when they move to a real cluster.

**Decision 2: Custom textarea/overlay syntax highlighter instead of Monaco for the primary shell.**
Why: The shell is an always-visible, always-mounted component embedded inside lesson cards. Loading Monaco for every lesson would add significant bundle weight (~2 MB) and introduce an async mount delay. The custom tokenizer is under 150 lines, zero dependencies, and renders synchronously in the same frame as the textarea.
Rejected alternative: Use `@monaco-editor/react` (already a dependency) for the shell itself.
Tradeoff: The custom tokenizer has no language server, no autocomplete, no bracket matching, and no error squiggles. It does handle the 130+ operators correctly for color, but edge cases like multi-line strings or nested regex flags can mis-tokenize. Monaco is used (per the dependency and `LESSON_VISUALIZERS` wiring) for richer editing contexts where the extra weight is justified.

**Decision 3: Model aggregation stages as static `{ type, spec }` objects with a presentational animation, not as a live execution engine.**
Why: Implementing a correct in-browser aggregation executor would require handling `$group` with accumulators, `$lookup` across in-memory collections, `$unwind` on nested arrays, and `$project` expression evaluation. That is a near-complete re-implementation of the MongoDB aggregation VM. The pedagogical goal of the visualizer is to show pipeline topology and data flow, not to produce correct output documents.
Rejected alternative: Execute each stage sequentially against the in-memory collections and show intermediate results between stages.
Tradeoff: The pipeline visualizer is honest about being a diagram tool, not a query runner. Learners who want to run pipelines must use the MongoShell — which only supports `find`, not `aggregate`. This is the most significant fidelity gap in the project.

## The hardest bug

The argument parser in `MongoShell` uses a regex substitution to convert unquoted keys to quoted JSON: `.replace(/(\w+)\s*:/g, '"$1":')`. This incorrectly quotes the `options` key in `$regex` objects, double-quoting already-quoted keys, and mangling values like `{ $options: "i" }` into `{ "$options": "i" }` which is still valid — but it also fires on things like `ISODate("2023-01-15")` if that string appears as a value, and it trips on `http://` literals where the `//` comment-stripping runs first. The fallback `new Function('return ' + argsStr)()` rescues most of these cases because it evaluates actual JavaScript, including regex literals and `new Date()`, bypassing the JSON normalizer entirely. The failure mode is silent: if both paths throw, the user sees "Could not parse query arguments" with the raw input echoed, but no line-level diagnostic. A learner typing a perfectly valid `{ name: /alice/i }` will sometimes hit this and be confused why it fails before the `new Function` path catches it on the second try.

## What you'd change at scale

The biggest architectural limit is that the query engine and pipeline visualizer are completely decoupled. The natural next step is implementing `aggregate` as an ordered reduce over the stage array, executing at least `$match`, `$project`, `$group`, `$sort`, `$limit`, `$unwind`, and `$addFields` against the in-memory collections. This would let MongoShell accept `db.orders.aggregate([...])` and show real intermediate documents at each stage, turning the pipeline visualizer from a diagram into a live debugger.

The lesson content model (raw HTML strings in JS objects) does not scale well past a few hundred lessons. Moving to MDX files with frontmatter would enable static validation, better diffs in version control, and tree-shakable imports so the initial bundle does not pay for all 178 lessons at once.

The B-tree visualizer uses a single hard-coded 23-value tree. A generative approach — building the tree from the actual sample data field values — would make it truthful. The current implementation is pedagogically correct (the traversal path logic is right) but a student who notices the fixed values may distrust it.

Progress is stored in `localStorage` via Zustand `persist`. There is no backend, no auth, no cross-device sync. For a paid course product this is the first infrastructure gap — a Supabase `user_progress` table keyed by `(user_id, course_id, module_id, lesson_id)` with a `status` and `score` column would take the current Zustand store shape and map it directly.

## Probing Q&A

**Q: How does your in-browser query engine actually execute `db.users.find({ age: { $gt: 25 } })`?**
A: MongoShell strips the input with the regex `db\.(\w+)\.(\w+)\(([\s\S]*)\)`, extracting `users`, `find`, and the args string. It normalizes the args to JSON (unquoted-key patch + JSON.parse, or new Function fallback), then calls `evaluateQuery('users', { age: { $gt: 25 } })`. That runs `collections.users.filter(doc => matchesQuery(doc, query))`. `matchesQuery` sees `age` is not a logical operator, resolves the field value from the document, recognizes `{ $gt: 25 }` as an operator object, and calls `matchesOperators(28, { $gt: 25 })`. The `$gt` case calls `comparable(28, 25)` (same type, passes) then `cmp(28, 25) <= 0` which is false, so the document passes the filter.

**Q: How does your in-browser query engine handle dot-notation like `{ "address.city": "San Francisco" }`?**
A: `matchesQuery` passes the key `address.city` to `getNestedValue(doc, 'address.city')`, which does `'address.city'.split('.').reduce((curr, k) => curr?.[k], doc)`, resolving `doc.address.city`. The value is then compared with the scalar condition using `looseEq`. Null-safety is handled: if any intermediate key is `undefined` or `null`, the reduce returns `undefined` and the equality check fails cleanly.

**Q: How faithful is the in-browser engine to real MongoDB semantics?**
A: Reasonably faithful for the covered operators. Key differences: (1) No BSON type coercion — JavaScript's `===` is used for equality, so `{ age: "28" }` will not match a document with numeric `age: 28`, which matches real MongoDB's typed equality. (2) `comparable()` rejects cross-type comparisons like `age > "25"`, consistent with BSON comparison order. (3) `$elemMatch` dispatches to `matchesQuery` for object element conditions and `matchesOperators` for operator-only conditions, matching the real dual behavior. (4) No `$text`, `$geoNear`, `$expr`, `$where`, or `$jsonSchema`. (5) The `aggregate` method is not supported at all — `db.orders.aggregate([...])` will throw "Method 'aggregate' is not supported."

**Q: How does the pipeline visualizer actually work? Is it executing the pipeline?**
A: No. The pipeline visualizer is a diagram with an animation. When you click "Visualize," `runAnimation` sets an `setInterval` at 800 ms that increments `activeStage` from 0 to `stages.length - 1`. The active stage gets a `border-mongo-bright shadow-glow` ring. No documents are filtered, grouped, or transformed. The input and output indicators show static counts (`inputDocs.length` and the string "transformed documents"). The default spec objects (e.g., `$group: { _id: '$category', total: { $sum: 1 } }`) are illustrative defaults, not computed from the sample data.

**Q: How is the B-tree index visualizer built, and is it accurate?**
A: It uses a static hand-coded tree: root `[30]`, internal nodes `[10,20]` and `[40,50]`, six leaf nodes. `simulateSearch(target)` deterministically computes which child at each level to highlight (root → `target <= 30` picks left child, then finds the bracket within that child's keys), builds a `path` array of three levels, and animates them at 500 ms steps. The traversal logic is accurate for the structure shown. The "found" determination checks whether `target` appears anywhere in the highlighted path nodes, so values on the path but not as a leaf key (e.g., the root value `30`) register as found. For COLLSCAN, it iterates through `allValues` (23 sorted integers) at 100 ms each, stopping when it finds the target, and shows docs-examined ticking up. The comparison is genuinely instructive: searching for `35` via IXSCAN examines 3 nodes, via COLLSCAN it walks through all values up to index 11. The limitation is the tree is not derived from real data — it is a fixed teaching example.

**Q: Why did you use a custom textarea overlay instead of Monaco for the shell?**
A: Bundle size and synchronous rendering. Monaco via `@monaco-editor/react` is roughly 2 MB and mounts asynchronously (lazy-loaded iframe worker). The shell appears in every lesson's Practice tab. A custom tokenizer that runs synchronously on every keystroke with no async overhead keeps the tab responsive and adds no perceptible load time. The tradeoff is no autocomplete, no error underlining, and potential mis-tokenization at edge cases.

**Q: How does your argument parser handle `db.users.find({ name: /alice/i })`?**
A: The first pass runs `argsStr.replace(/(\w+)\s*:/g, '"$1":')` which produces `{ "name": /alice/i }`, which is not valid JSON, so `JSON.parse` throws. The catch block then runs `new Function('return ' + argsStr)()`, which evaluates the original string as JavaScript, producing a real `RegExp` object `{ name: /alice/i }`. In `matchesQuery`, when the condition is a `RegExp` instance (detected by `condition instanceof RegExp`), it calls `condition.test(value)` directly. So regex literals work correctly through the fallback path.

**Q: How is progress persisted and what are its limits?**
A: Zustand's `persist` middleware serializes the store to `localStorage` under the key `mongo-mastery-storage`. The stored shape is `{ progress: { "courseId.moduleId.lessonId": { status, practiceScore, updatedAt } }, bookmarks: [...], shellHistory: [...] }`. Shell history is capped at 100 entries (`.slice(-99)`). Limits: single device, cleared on browser data wipe, no user identity, no cross-device sync. There is no backend.

**Q: Where exactly does the `$type` operator map BSON type numbers to string aliases?**
A: In `sampleData.js`, `normalizeType(t)` maps numeric codes to strings: `1 → 'double'`, `2 → 'string'`, `3 → 'object'`, `4 → 'array'`, `7 → 'objectId'`, `8 → 'bool'`, `9 → 'date'`, `10 → 'null'`, `16 → 'int'`, `18 → 'long'`, `19 → 'decimal'`. `bsonType(value)` infers the JS-runtime type: `Date` → `'date'`, `null` → `'null'`, `Array.isArray` → `'array'`, `Number.isInteger` → `'int'`, non-integer number → `'double'`, etc. `matchesType` accepts an array of type aliases or numbers, normalizes them, and checks the inferred type — with a special case where `'number'` matches any of `int`, `double`, `long`, `decimal`.

**Q: The `evaluateQuery` function only supports `find`-style filter matching. How would you add real aggregation support?**
A: Wire a new `aggregate` method in `executeMethod` that takes an array of stage objects and runs a sequential reduce: `stages.reduce((docs, stage) => executeStage(docs, stage), collections[collName])`. Each stage handler receives the current document set and returns a new one. `$match` runs the existing `matchesQuery` filter. `$project` maps each document to a subset of fields. `$group` uses a Map keyed by the `_id` expression value, accumulating `$sum`, `$avg`, `$min`, `$max`, `$push` per group. `$sort` calls `Array.sort` with a comparator derived from the sort spec. `$limit` and `$skip` are `slice` calls. `$unwind` flatMaps each document into N documents per array element. The hard parts are expression evaluation (e.g., `{ $concat: ['$first', ' ', '$last'] }`) and `$lookup` joins across collections — both require an expression interpreter, which is the same scope problem the current engine avoids by limiting to filter-only queries.
